Database Schema

Schema Design Decisions

  • Enum types for sport_type visibility friendship_status — DB-level validation
  • ON DELETE CASCADE on all FK — no orphaned rows
  • GEOMETRY(LineString, 3857) — Web Mercator so distances are in metres
  • UNIQUE(segment_id, activity_id) — one effort per activity per segment
  • UNIQUE(requester_id, addressee_id) — no duplicate friend requests
  • activity_athletes — pure join table, composite PK, no extra id needed

Main Feature: Route Matching

If two athletes run the same route — they appear in a shared leaderboard.

Uses Fréchet Distance — measures similarity between two curves.

densifyFrac=0.05 adds intermediate points so short polylines don't get false negatives. 
SELECT u.username, a.title,
  TO_CHAR((a.duration||' seconds')
    ::INTERVAL, 'MI:SS') AS time,
  ROUND(ST_FrechetDistance(
    ca.path, a.path, 0.05)::numeric)
    AS route_deviation_m,
  RANK() OVER (
    PARTITION BY u.id
    ORDER BY a.duration)   AS personal_rank
FROM common_activities ca
JOIN activities a
  ON a.common_activity_id = ca.id
JOIN users u ON u.id = a.owner_id
WHERE ca.id = 1001
  AND a.duration IS NOT NULL
ORDER BY a.duration;

Segments & Leaderboard

Timed segments on a route. Athletes compete for the fastest time.

  • Each segment has a path geometry
  • Effort recorded per activity
  • RANK() window function over best times
SELECT
  u.username,
  MIN(se.elapsed_time) AS best_time_sec,
  TO_CHAR(
    (MIN(se.elapsed_time)||' seconds')
    ::INTERVAL, 'MI:SS')  AS best_time,
  RANK() OVER (
    ORDER BY MIN(se.elapsed_time)) AS rank
FROM segment_efforts se
JOIN users u ON u.id = se.athlete_id
WHERE se.segment_id = 3001
GROUP BY u.id, u.username
ORDER BY best_time_sec;

Feed with Access Control

Three visibility levels: public friends private

Friendship is bidirectional — stored as one row, queried in both directions with UNION. 
SELECT a.id, a.title, u.username,
  ROUND((a.distance/1000.0)::numeric,2)
    AS distance_km,
  COUNT(k.id) AS kudos
FROM activities a
JOIN users u ON u.id = a.owner_id
LEFT JOIN kudos k ON k.activity_id = a.id
WHERE a.owner_id = 1001 OR (
  a.visibility = 'public' AND
  a.owner_id IN (
    SELECT addressee_id FROM friendships
    WHERE requester_id = 1001
      AND status = 'accepted'
    UNION
    SELECT requester_id FROM friendships
    WHERE addressee_id = 1001
      AND status = 'accepted'))
GROUP BY a.id, u.username,
  a.distance, a.duration, a.started_at
ORDER BY a.started_at DESC;

Indexes & Performance

IndexTypeWhy
activities(owner_id)B-treeFeed filters by owner every request
activities(started_at)B-treeFeed sorted by date — avoids full sort
activities(visibility)B-treeAccess control filter on every query
activities(path)GISTSpatial search for Fréchet matching
common_activities(path)GISTFind nearby reference routes
segments(path)GISTSegment detection on activity upload
friendships(requester_id/addressee_id)B-treeBidirectional friend lookup
segment_efforts(segment_id/athlete_id)B-treeLeaderboard joins
GIST = R-tree for geometry. Standard B-tree can't index spatial data — GIST builds bounding-box hierarchy for fast proximity filtering before computing exact Fréchet distance.
